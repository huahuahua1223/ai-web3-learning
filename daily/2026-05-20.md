## 问题与收获

**问题 1**：直播里 draken 老师讲 Hermes 的 skills 机制时，我没完全 catch 到 skills 是怎么被 Agent 主动调用的——是靠语义匹配还是用户在 prompt 里显式 cue？读 install guide 的 `hermes config edit` 那段也没有正面答案。明天装完一定要在 TUI 里试一个 skill 看实际触发链路。

**问题 2**：受限 Web3 助手任务里，我犹豫了很久「分析交易 vs 模拟交易」这一层要不要分开——分析是「告诉用户这笔可能亏」，模拟是「告诉用户这笔执行后链上状态会怎样」。最后还是分开了，但意识到现实里很多 dApp 把这两步揉成一个「Preview」按钮，可能因为对用户来说不需要那么细粒度。

**收获 1**：Hermes 定位的更精确表述——从 5/18 写过的「长期运行的个人自动化助手」推进到「**有 PID、占端口、用我自己的 Key、能被 `ps` 看到的进程**」。这个差别在写代码层面看是 SaaS vs 本地服务的区别，在认知层面是「Agent 的概念」vs「Agent 的我」的区别。

**收获 2**：把每日打卡 SOP + WCB API 速查写进 `CLAUDE.md` 的工程化反思——以前每天打卡靠回忆 + 翻 WCB 网页，现在 `scripts/wcb-checkin-prep.sh` 一键拉齐 WCB 进度 + 残酷共学最近 3 天 + 本地 commit。Agent 助手的可复用性 = **把流程沉淀成脚本而不是上下文**。上下文每次新对话都会重置，脚本不会。

**收获 3**：「读了一遍 install guide 但没装」这件事本身的观察——以前会觉得「读完就懂了」，今天意识到读懂 install 步骤和真的让 Agent 进自己电脑是两回事。这个延迟让我看到一件事：**对 Agent 真正的判断，要在它真正占着我电脑的端口、消耗着我申请的 key、回复着我发的消息之后才能给**。
