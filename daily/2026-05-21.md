#### Handbook：AI > Agent

完成首轮通读。

**章节定义**:Agent = **被约束的执行循环**,五要素缺一不可——目标、工具、状态、权限、停止条件。关键不在"像人",在"明确权限和可审计流程里把建议推进到行动"。

**章节五个知识节点**:Tool Use / Planning / **State**(可查询、可恢复、可审计;不能只藏在模型上下文里)/ Reflection / Multi-Agent

**跟我自己的实操对照**:

| 我做过的事 | Handbook 标准术语 | 对照 |
|---|---|---|
| 5/19 让 Claude Code 拿 WCB Key 调 `users.getProfile` | **Tool Use**(只读工具)| 完全对齐;Handbook 把只读和写入工具显式分开 |
| 跑 `curl → jq → git` 流水线 | **Planning** + 多步 Tool Use | 我直觉地拆成了步骤,Handbook 给这个直觉命名了 |
| state 只放在 prompt 历史里(每次新会话重置)| **State 必须外置**(可查询、可恢复、可审计)| **Handbook 明确说"只放 prompt 历史不够"** — 这就是 5/20 我自己琢磨的"脚本沉淀 > 上下文记忆"的更系统版本 |

**让我印象最深的 3 个点**:

1. **"自我反思仍由模型完成,模型可能给自己错误找理由 …… 自我检查可以提高质量,确定性检查才能承载风险"** —— 这一句直接颠覆了我对"加一个 reflection 步骤就能提质量"的乐观。Reflection 不能当最后一道闸,**质量靠 reflection,风险靠 simulation / policy / human / log**(确定性检查)。
2. **"最危险的设计是让 Agent 同时拥有模糊目标、广泛工具、长期记忆和大额资产权限"** —— 这恰好是 Hermes(我 5/22 要装的)会逐步靠近的位置:长期运行 + skills 沉淀 + 跨会话记忆。装的时候要时刻警惕,**不要把它放在能动钱的位置上**。
3. **AI × Web3 Agent 8 步标准架构**:用户目标 → 读上下文 → 生成计划 → 拆只读/候选写入 → 只读自动执行 → 写入进入 policy 检查 → Simulation → 用户确认 → Wallet/Smart Account 执行 → 日志。**这就是我 5/19「受限 Web3 助手设计」的更标准展开**(详见下一节的对照)。

**修正自己的术语误用**:之前我以为 Handbook 用"ReAct 循环"这个词,实际章节用的是**独立的三个节点**——Planning(拆步骤)、State(外置可查)、Reflection(中间结果自检)。ReAct 是论文术语,Handbook 把它**拆得更细更可工程化**,把单一概念分成三个独立的工程关注点。

#### Handbook：Bridge > Web3 Tool Use

完成首轮通读。

**章节定义**:把 RPC / 合约 / 钱包 / explorer / DeFi 操作变成 Agent 可调用工具,**真正难的不是"能调用",而是「权限、参数、模拟、日志」**。

**章节 8 类工具节点**(按风险升序):RPC Tool → Contract Read → Contract Write → Wallet Tool → Explorer Tool → DeFi Tool → **Tool Permission** → **Tool Log**

**核心原则三条**:① **读写分离**(读和写必须是不同工具、不同权限,不能合一个"万能 RPC")② **参数结构化**(`chain id / contract address / method / args / value / slippage` 不能埋在自然语言里)③ **日志不可省**(每次调用都要记录输入、输出、时间、来源、错误)

**跟我 5/19「受限 Web3 助手设计」的对照(关键!)**:

| 我设计的四层 | Handbook 的对应表达 | 差异 / 修正方向 |
|---|---|---|
| 信息层(免审)| **RPC Tool / Contract Read / Explorer Tool**(只读工具集合,自动允许)| 对齐 ✅,但 Handbook 拆得更细——同样是只读,RPC 和 Explorer 各有不同用途 |
| 分析层(用户复核)| **生成交易草稿:自动允许**(不发送)| Handbook 比我更准——草稿可以自动生成,不需要复核,**复核动作放在签名前**而不是生成时 |
| 模拟层(强提示)| **Simulation 不是独立权限层,是"写交易前的必经 gate"** | **关键认知更新**:我把它当层,Handbook 把它当 gate(每笔写交易都过)——这是工程上的本质区别 |
| 签名层(硬卡死)| **拆成三段**:① 小额白名单 — session key 允许 / ② 大额转账或授权 — 必须人工确认 / ③ 任意合约调用 — 默认禁止 | **Handbook 比我多一层 session key 中间档** — 完全人工确认会让 Agent 失去自动化价值,完全放开又危险,session key + 额度白名单是工业级中间方案 |

**让我印象最深的 3 个点**:

1. **session key 是我没想到的层**。我之前的设计直接从"模拟"跳到"签名硬卡死",中间没有"小额自动化通道"。Handbook 给出五级 Tool Permission 分层:**查询余额(自动)→ 生成交易草稿(自动)→ 小额白名单(session key)→ 大额(人工)→ 任意合约(默认禁止)**。session key 这一层是 Agent 实用化的核心。
2. **"读写分离"被写成铁律**。我做 WCB API 时把 `users.getProfile`(query)和 `tasks.submitEvidence`(mutation)分开看是直觉,Web3 场景把这种直觉**变成了强制规则**——因为读错只是误导,写错改变资产。
3. **工具调用日志的完整字段**:`用户目标 / 工具名 / 输入参数 / 输出结果 / 错误 / 时间 / 链 ID / 区块号 / 交易哈希 / 确认人 / policy 判断`。我之前做 WCB API 时只记了"调了哪个 procedure",**"确认人"和"policy 判断"这两个字段是新的关键认知**——这两个字段决定了出问题时能不能回溯责任。

**问题 2(原版)直接被章节回答了**:我原来问"签名层硬卡死用什么机制实现"——Handbook 给的答案组合是 **session key + Smart Account policy + simulation + 大额人工确认**(章节没展开具体协议如 ERC-7715 / ERC-4337,但给了术语骨架)。这一条从"问题"升级为"已解锁认知",顺势成为修订 `restricted-web3-assistant-design.md` 的依据(详见明日计划)。

## 问题与收获

**问题 1**:Handbook 明确说"**自我反思仍由模型完成,模型可能给自己错误找理由**"——那 Multi-Agent 里让"安全 Agent"做风险审查是不是仍有同质问题?(只是把"模型给自己找理由"变成"另一个模型给前一个模型找理由")"两个模型的对话"是否本质上还是"一个分布的自检",不构成确定性检查?**明天怎么验证**:在 5/22 Week 1 Review 现场提问,或者翻 OpenAI Agents Guide 看 Multi-Agent 跟 Guardrails 是怎么区分的——guardrails 应该是确定性的,而 multi-agent 互检不是。

**问题 2**:Handbook 给了 session key / Smart Account policy / simulation 这套术语骨架,但没展开**具体协议(ERC-7715 / ERC-4337 / EIP-7702)是怎么实现 session key 的**。明天怎么验证:Week 2 备战时读 Bridge > Agent Wallet + Web3 > Account Abstraction 章节,把"session key 这个能力具体落在哪个 EIP 上"搞清楚。

**收获 1**:**先做后读 > 先读后做**(今日核心节奏论)。一周轨迹:5/18 贴标签 → 5/19 实操让 Agent 调 WCB API → 5/20 看直播+install guide 没装 → 5/21 读 Handbook。原以为"今天才认真读 Handbook"是落后,实际效果是反过来的——动手过的概念再读文档,理解从"建模"变成"命名",每一句术语都能找到自己脑子里已存在的实操记忆当锚点。这个节奏不该被否定,它就是我的学习节奏。

**收获 2**:**Handbook 的价值在于"给直觉命名 + 暴露盲区"**。我之前以为读 Handbook 是"获取新知识",今天意识到对一个已经动过手的人,Handbook 真正的两个价值是:① **给直觉命名**(我做 WCB API 时把 query/mutation 分开是直觉,Web3 场景把这变成铁律"读写分离");② **暴露盲区**——我设计的「四层风险」里把"模拟"当层,Handbook 给的是"Simulation 是写交易前的必经 gate",这是我之前没看见的工程区别;**我的设计缺了"session key 小额白名单"这一层**,Handbook 直接补上来了。

**收获 3**:**"自我反思仍由模型完成,模型可能给自己错误找理由 …… 确定性检查才能承载风险"**——这是今天单独立项的一个认知颠覆。我之前一直以为"加一道 reflection 步骤就能提质量",今天意识到 reflection 只能管"质量",**不能管"风险"**。风险必须靠 simulation / policy / human-in-the-loop / log 这些**确定性的、跟模型无关的**机制来兜。这给我一个直接的工程原则:**任何一道闸,如果它能被模型说服关掉,它就不是真的闸**。
